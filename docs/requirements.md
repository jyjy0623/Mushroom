## 一、产品概述

### 1.1 产品定位

"蘑菇大冒险"是一款面向学生（主要为中小学生）的学习激励与成长记录 Android 应用。通过游戏化的"蘑菇奖励"体系，将枯燥的学习任务转化为有趣的成就收集过程，帮助孩子建立良好的学习习惯，同时让家长参与到奖励兑换的互动中。

### 1.2 目标用户

| 用户角色 | 描述 |
|---------|------|
| 主用户（学生）| 中小学生，完成每日任务、记录学习进度、收集蘑菇奖励 |
| 辅助用户（家长）| 设定任务、审核打卡、管理奖品库、确认奖励兑换 |

### 1.3 核心价值

- **激励驱动**：用蘑菇积分将学习成果可视化，增强成就感
- **习惯养成**：每日任务打卡帮助建立规律学习习惯
- **目标管理**：里程碑体系让长期学习目标可拆解、可追踪
- **亲子互动**：奖品兑换环节促进家长与孩子的正向互动

### 2.2 任务管理模块

#### 2.2.1 每日任务设定（V1）

**功能描述**：家长或学生可以手动创建每日学习任务，支持按学科、类型分类。

**需求细节**：
- 支持添加、编辑、删除任务
- 任务字段：任务名称、学科分类、预计时长、任务描述、截止时间（deadline）
- **截止时间为可选字段**，设定后作为该任务的完成期限基准
- 支持任务复制（将某天任务复制到其他天）
- 支持设置"重复任务"（每天/工作日/自定义周期）
- 任务状态：待完成 / 提前完成 / 准时完成 / 已跳过

**学科分类预设**：数学、语文、英语、物理、化学、生物、历史、地理、其他

#### 2.2.1a 新建任务页模板选择器（V1）

**功能描述**：在新建任务页面，提供快速从模板填充任务信息的入口，减少重复输入。

**位置**：任务日期字段下方、任务名称字段上方（仅新建任务时显示，编辑/只读时隐藏）

**交互设计**：
- 下拉控件，标签为「任务模板」
- 默认选中「（不使用模板）」，不触发任何填充
- 下拉列表分两组，组标题不可选择：
  - **系统内置**：列出所有 `isBuiltIn = true` 的模板
  - **自定义**：列出所有 `isBuiltIn = false` 的模板（若为空则不显示该组）
- 选中某模板后，自动填充以下字段（覆盖当前内容）：
  - 任务名称（title）
  - 学科分类（subject）
  - 预计时长（estimatedMinutes）
  - 任务说明（description）
  - 截止时间（deadline）：从模板的 `defaultDeadlineOffset`（分钟数）推算，以新建当天日期为基准（无偏移量则不设截止时间）
  - 完成奖励 / 提前完成奖励：从模板 `rewardConfig` 填充，同时开启「自定义奖励」开关
- 模板选择**不持久化**到任务记录，用户仍可手动修改填充后的各字段
- 选中模板后，下拉框保持显示所选模板名称，不因字段修改而联动变化

#### 2.2.1b 新建任务/里程碑合并入口（V1）

**功能描述**：将"新建任务"与"新建里程碑"合并为同一个页面，通过 Tab 切换，降低操作层级，减少导航跳转。

**页面标题**：新建任务或里程碑

**布局结构**：
- 顶部共享区域：「日期」字段（只读文本框 + 修改按钮），两个 Tab 共用，修改后同步影响任务日期和里程碑的 scheduledDate
- TabRow：Tab[0]「任务」，Tab[1]「里程碑」（仅点击切换，禁用左右滑动手势）
- HorizontalPager：各 Tab 内容独立滚动

**Tab[0]「任务」内容**（与原新建任务页一致，去掉独立的日期字段）：
- 任务模板（下拉选择器）
- 任务名称 *、任务说明、学科、预计时长、截止时间、重复规则
- 奖励设置（Card）
- 另存为自定义模板（OutlinedButton）
- 顶部栏右侧显示「保存」按钮

**Tab[1]「里程碑」内容**（与原新建里程碑页一致，去掉独立的日期字段）：
- 名称 *、类型（下拉）、学科（下拉）
- 分数段奖励配置（Card）
- 页内全宽「保存里程碑（需家长确认）」Button
- 顶部栏右侧无按钮（里程碑通过页内按钮保存）

**触发入口**：
- 任务列表右下角「+」→ 进入合并页，默认停在 Tab[0]「任务」
- 任务列表「+」FAB 展开菜单中的「添加里程碑」→ 进入合并页，默认停在 Tab[1]「里程碑」
- 里程碑列表「新建」→ 仍导航至独立 `MilestoneEditScreen`（用于编辑现有里程碑的场景保持不变）

**编辑任务/只读模式**：合并页仅用于新建，编辑和只读继续使用原 `TaskEditScreen` 逻辑（`taskId != null` 时顶部标题为"编辑任务"，无 Tab）

#### 2.2.2 任务模板管理（V1）

- 支持将常用任务组合保存为模板
- 支持一键应用模板到某天或某周
- 系统内置以下**预设奖励任务模板**，家长可直接启用，也可修改奖励参数：

##### 预设模板一：早晨晨读

| 字段 | 内容 |
|-----|------|
| 模板名称 | 早晨晨读 |
| 学科分类 | 语文 / 英语（可选） |
| 预计时长 | 15～30 分钟 |
| 触发条件 | 在指定晨读时间段内（如 7:00～7:30）完成打卡 |
| 默认奖励 | 小蘑菇 ×1 |
| 按时晨读加奖 | 在时间段开始后 10 分钟内打卡开始，额外获得小蘑菇 ×1（奖励"雷打不动"的好习惯） |
| 备注 | 打卡时可附加朗读内容截图或文字记录 |

> 设计说明：晨读是培养语言感知力和专注力的重要习惯，门槛设低（小蘑菇×1）保证每天都能得到即时正反馈，而"按时开始"的额外奖励则进一步强化准时起床、主动学习的习惯。

##### 预设模板二：作业完整记录在备忘录

| 字段 | 内容 |
|-----|------|
| 模板名称 | 作业记录打卡 |
| 学科分类 | 通用（跨学科） |
| 预计时长 | 5 分钟 |
| 触发条件 | 每天放学后，在 APP 内完整录入当天所有学科的作业清单并提交 |
| 默认奖励 | 小蘑菇 ×1 |
| 连续记录加奖 | 连续 5 天均完整记录，额外获得中蘑菇 ×1（奖励坚持的习惯） |
| 备注 | 需记录每科作业名称及预计完成时间；家长可开启"审核模式"核验记录完整性 |

> 设计说明：养成记录作业的习惯是时间管理的基础技能。此模板门槛极低（5分钟），但对习惯养成价值极高。连续5天奖励设计让孩子有短期阶段目标，避免只坚持一两天就放弃。

##### 预设模板三：在校完成作业

| 字段 | 内容 |
|-----|------|
| 模板名称 | 在校完成作业 |
| 学科分类 | 通用 |
| 预计时长 | — |
| 触发条件 | 当天放学时（如17:30前）打卡确认，并注明在校已完成的作业科目 |
| 默认奖励 | 中蘑菇 ×1 |
| 全部在校完成加奖 | 当天所有作业均在校完成，额外获得中蘑菇 ×1 |
| 备注 | 需附上在校完成的作业列表；家长可要求上传作业照片作为凭证 |

> 设计说明：在校完成作业是高效学习的优质习惯，回家后有更多时间进行自主阅读或兴趣活动。奖励等级设为中蘑菇（高于日常任务的小蘑菇），体现对这一高质量习惯的重视。

#### 2.2.3 AI 智能任务生成（V2，架构预留）

- 通过语音输入描述学习目标，调用大模型自动生成结构化任务
- 大模型接口抽象为 `TaskGeneratorService`，V1 提供手动输入实现，V2 替换为 AI 实现

### 2.3 打卡记录模块

#### 2.3.1 每日打卡

**功能描述**：学生完成任务后进行打卡确认，记录当天学习情况。

**需求细节**：
- 任务列表中每项任务有"完成打卡"按钮
- 打卡时可附加备注（文字/图片）
- 打卡时系统自动记录实际完成时间戳
- 支持家长审核打卡（可选开启）
- 全天任务全部完成后触发"全勤奖励"
- 支持补打卡（当天24:00前）

#### 2.3.2 提前完成奖励

**功能描述**：针对设置了截止时间的任务，若学生在截止时间之前提交打卡，系统自动识别并给予额外奖励，激励高效学习。

**需求细节**：
- 仅对**设置了截止时间（deadline）的任务**生效
- 打卡提交时，系统比较"提交时间"与"截止时间"，若提交时间早于截止时间则判定为"提前完成"
- 提前完成的任务在任务列表中以特殊标识显示（如闪电图标 ⚡）
- 提前完成额外获得奖励蘑菇（奖励类型和数量可由家长配置，默认：额外获得 1 个小蘑菇）

> 以上分档和奖励数量均可由家长在设置中自定义，也可关闭该功能（退回到不区分提前/准时的模式）。

- 补打卡（截止时间之后提交）**不享受**提前完成奖励
- 打卡历史中记录每项任务是否为"提前完成"，并展示提前时长

#### 2.3.2 打卡历史

- 日历视图展示打卡记录（不同完成率显示不同颜色深度）
- 支持查看某天的详细打卡记录
- 显示连续打卡天数（streak）

#### 2.4.3 关键奖励时间

- 支持设置关键时间的奖励条件（如连续打卡N天、测验达到X分）

#### 2.4.4 蘑菇扣分机制

**设计原则**：扣分是对不良习惯的温和约束，而非惩罚。扣分规则需由家长主动开启并配置，默认关闭，且扣分数量始终低于同类行为的正向奖励，确保整体体验正向为主。扣分后蘑菇余额最低为 0，不会出现负数。

##### 扣分触发方式

扣分事件由**家长手动记录**（非系统自动检测），家长在发现不良行为后进入"扣分记录"页面选择扣分项并提交，系统从蘑菇余额中扣除对应数量。每次扣分均记录在蘑菇账本中，标注扣分原因，账目透明。

##### 预设扣分模板

系统内置以下预设扣分项，家长可按需开启，并可自定义扣分数量和名称：

| 扣分项 | 默认扣分 | 扣分蘑菇类型 | 触发条件说明 |
|-------|---------|------------|------------|
| 忘记带作业 | 小蘑菇 ×2 | 小蘑菇 | 上学时忘记带某科作业本 |
| 没有订正试卷 | 小蘑菇 ×3 | 小蘑菇 | 考试/测验后规定时间内未完成错题订正 |
| 作业未记录在备忘录 | 小蘑菇 ×1 | 小蘑菇 | 当天放学后未完整记录作业清单 |
| 作业未完成就玩耍 | 中蘑菇 ×1 | 中蘑菇 | 未完成当天作业擅自进行娱乐活动 |
| 上课迟到 | 小蘑菇 ×2 | 小蘑菇 | 无故上课迟到 |

> 家长可在"设置 → 扣分规则"中开启/关闭单个扣分项，并调整扣分数量。初次使用建议只开启1～2项，避免规则过多导致孩子产生挫败感。

##### 扣分设计约束

- **扣分上限**：单次最多扣除中蘑菇 ×1 或小蘑菇 ×5，防止一次扣分打击孩子积极性
- **余额保底**：蘑菇余额最低为 0，不出现负分
- **申诉机制**：学生可对扣分记录发起申诉，家长审核后可撤销该次扣分
- **扣分展示**：蘑菇账本中扣分记录以红色标注（区别于绿色的加分记录），账目清晰透明
- **每日扣分汇总**：首页当日扣分情况以单独区域提示，避免与奖励混淆
- **不可重复扣分**：同一事件同一天只能被记录一次扣分，防止重复惩罚

##### 扣分与正向激励的平衡建议

扣分机制的目的是帮助孩子建立规则意识，而非打压积极性。建议家长：
1. 扣分后及时与孩子沟通原因，而非仅依赖 APP 传递信息
2. 扣分项数量控制在 3 项以内，集中在最需要改善的习惯上
3. 每周末复盘时，若孩子整周无扣分，可额外给予"零失误奖励"（小蘑菇 ×2）

#### 2.5.1a 奖品列表页 UI 规范（v1.5 更新）

**整体布局**：单列列表（非双列 Grid），去掉图标占位块，每个奖品占一行，避免对齐问题。

**分区结构**：

| 区域 | 条件 | 默认状态 |
|------|------|---------|
| 进行中（N） | `status == ACTIVE` | 展开 |
| 已完成（N） | `status == COMPLETED 或 CLAIMED` | 折叠 |

**进行中区域——每行格式**：
```

```
- 左：奖品名称（bodyMedium，加粗）
- 右：进度文字（labelSmall，灰色）

- 时长型：只显示「已用 X / 上限 Y 分钟」

**已完成区域**：
- 折叠态：标题行「已完成（N）▶」，点击展开
- 展开态：每行显示奖品名称 + 「已兑换」灰色标签，无进度条

**其他**：
- 保留顶部 AppBar 和右下角 FAB（创建奖品）
- 去掉封面图片和 emoji 图标占位 Box
- 点击任意行仍导航到详情页

#### 2.5.2 拼图积累展示

**核心交互**：
- 每件奖品以"拼图"方式展示兑换进度
- 家长设置所需总积分（如 100 分）和拼图总块数（如 20 块），系统自动计算每块门槛（5 分/块）
- 兑换时学生选择蘑菇类型和数量，按等级折算积分，系统自动计算可解锁拼图块数
- 拼图全部解锁后，奖品可领取，动画庆祝效果展示完整图片

**积分折算规则**：
| 蘑菇等级 | 兑换积分 |
|---------|---------|
| 小蘑菇 | 1 分/个 |
| 中蘑菇 | 5 分/个 |
| 大蘑菇 | 25 分/个 |
| 金蘑菇 | 100 分/个 |
| 传说蘑菇 | 500 分/个 |

**示例**：
```
高达模型（所需总积分：100分，拼图总块数：20块）
每块门槛：100 ÷ 20 = 5 分/块

兑换操作：中蘑菇 × 3 = 15 分 → 解锁 3 块
当前进度：已解锁 15/20 块 ███████████████░░░░░ 75%
```

#### 2.5.3 兑换流程

1. 学生在奖品页面选择想要的奖品
2. 查看当前拼图进度及每块所需积分
3. 选择蘑菇类型和数量，实时预览"贡献积分 X 分 → 可解锁 Y 块"
4. 确认兑换，蘑菇扣减，拼图块更新
5. 拼图完成时触发庆祝动画，奖品标记为"待领取"
6. 家长线下兑现奖品后标记"已领取"

### 2.7 数学试点奖励方案

> 本方案为数学学科试点的具体奖励设计，以"看电视 / 玩游戏 / 高达模型"三类奖品为例，体现**日常小奖励高频触达、游戏奖励克制适量、大奖励长期目标激励**的设计原则。该方案作为系统预设模板内置，家长可在此基础上调整参数。

#### 2.7.1 奖励分级设计原则

| 奖励类型 | 定位 | 兑换门槛 | 频次控制 | 目的 |
|---------|------|---------|---------|------|
| 看电视 | 日常消遣，正向反馈 | 低（小蘑菇） | 每周可兑换，有时长上限 | 保持日常学习积极性 |
| 玩游戏 | 适度娱乐，克制为主 | 中（中/大蘑菇） | 每月上限次数，防止成瘾 | 作为短期努力的阶段奖励 |
| 高达模型 | 长期大目标 | 高（金/传说蘑菇） | 与里程碑绑定，无频次限制 | 激励长期坚持和大考冲刺 |

#### 2.7.2 看电视奖励方案

**定位**：平时消遣，触手可得，让孩子感受到日常努力的即时回报。

**兑换规则**：

| 兑换内容 | 所需蘑菇 | 获得时长 | 每周上限 |
|---------|---------|---------|---------|
| 看电视 30 分钟 | 小蘑菇 ×3 | 30分钟 | 3次（共90分钟/周） |
| 看电视 60 分钟 | 小蘑菇 ×5 | 60分钟 | 1次（含在90分钟内） |

**蘑菇获取节奏（数学科目日常）**：
- 每天完成数学作业打卡：小蘑菇 ×1
- 提前完成：额外小蘑菇 ×1～2
- 每日全勤：中蘑菇 ×1（相当于5个小蘑菇）
- 预计每周可自然积累：**小蘑菇 7～14 个**
- 兑换一次30分钟看电视（3个小蘑菇）门槛很低，**约2～3天即可兑换一次**，保持高频正反馈

**约束条件**：
- 每周看电视总时长上限为 **90分钟**（由家长设置，可调整）
- 超出上限后当周无法继续兑换看电视
- 已兑换的时长需在当周内使用，不可累计到下周

#### 2.7.3 玩游戏奖励方案

**定位**：适度娱乐，比看电视门槛更高，每次奖励需要阶段性积累，防止孩子为了游戏而游戏。

**兑换规则**：

| 兑换内容      | 所需蘑菇   | 获得时长 | 每月上限        |
| --------- | ------ | ---- | ----------- |
| 玩游戏 30 分钟 | 中蘑菇 ×2 | 30分钟 | 2次（共60分钟/月） |
| 玩游戏 60 分钟 | 大蘑菇 ×1 | 60分钟 | 1次（含在60分钟内） |

**蘑菇获取节奏（需要阶段积累）**：
- 中蘑菇获取途径：每日全勤、过程小测验达标
- 大蘑菇获取途径：连续7天全勤、每周自测达标
- 兑换一次30分钟游戏（中蘑菇×2）需要**约2周每日全勤**
- 兑换一次60分钟游戏（大蘑菇×1）需要**连续7天全勤 或 每周自测达标**
- 门槛设计确保游戏奖励是**努力后的阶段性犒劳**，而非随时可得

**约束条件**：
- 每月玩游戏总时长上限为 **60分钟**（家长可调整）
- 单次游戏时长最长 60 分钟，不可拆分使用
- 超出每月上限后当月无法兑换游戏时间
- 每次游戏兑换需**家长确认**（额外提示游戏上瘾风险）
- 家长可设置"游戏冷却期"：两次游戏兑换之间至少间隔 7 天

#### 2.7.4 高达模型奖励方案

**定位**：长期大目标，与学科里程碑深度绑定，孩子需要持续努力数月才能集齐，具有强烈的成就感和仪式感。

**拼图设计**：

以"钢铁骑士高达 MG 1/100"为例：

### 2.9 版本自动检测与升级提示

#### 2.9.1 功能概述

应用在每次启动时（或进入主界面后）自动在后台检测是否有新版本可用，如有新版本则以对话框方式非打扰地提示用户，用户可选择立即前往下载或稍后提醒。

#### 2.9.2 版本信息来源

| 方案 | 说明 | 适用场景 |
|------|------|---------|
| **GitHub Releases API（推荐，V1采用）** | 读取 GitHub 仓库的 Latest Release 信息，获取最新版本号和更新日志 | 开源/自托管发布，无需额外服务器 |
| 自定义更新服务器 | 访问自有服务端 JSON 接口 | V2 扩展，支持灰度发布、强制升级 |
| Google Play In-App Update | 调用官方 Play 应用内更新 API | 通过 Play 商店分发时使用 |

**V1 采用 GitHub Releases API**：请求 `https://api.github.com/repos/{owner}/{repo}/releases/latest`，解析响应中的 `tag_name`（版本号）和 `body`（更新日志）。

#### 2.9.3 版本比较规则

- 应用当前版本从 `BuildConfig.VERSION_NAME` 读取，格式为语义化版本（`MAJOR.MINOR.PATCH`，如 `1.2.0`）
- 远端版本 tag 格式为 `v{MAJOR}.{MINOR}.{PATCH}`（如 `v1.2.0`），去掉前缀 `v` 后按三段整数比较
- 比较规则：`remoteMajor > localMajor`，或主版本相同且 `remoteMinor > localMinor`，或前两段相同且 `remotePatch > localPatch`，判定为有新版本
- 版本号格式非法（网络异常、JSON 解析失败等）时，静默忽略，不展示任何提示

#### 2.9.4 交互设计

**触发时机**：
- App 冷启动后，主界面（`DailyTaskListScreen`）加载完成时，在后台发起检测
- 不阻塞启动流程，检测在 IO 协程中进行
- 同一版本的更新提示，每次 App 启动最多弹出一次（用内存标记，不持久化到磁盘）

**提示 UI — 更新弹窗**：

```
┌─────────────────────────────────┐
│  🍄 发现新版本 v1.2.0           │
├─────────────────────────────────┤
│  当前版本：v1.0.0               │
│                                  │
│  更新内容：                      │
│  • 新增拼图动画效果              │
│  • 修复打卡记录偶发丢失问题      │
│  • 性能优化                      │
├─────────────────────────────────┤
│  [稍后提醒]        [前往下载]    │
└─────────────────────────────────┘
```

- **标题**：🍄 发现新版本 `{remoteVersion}`
- **当前版本**：显示本地 `BuildConfig.VERSION_NAME`
- **更新内容**：展示 GitHub Release 的 `body` 字段（Markdown 纯文本，取前 300 字）
- **[前往下载]**：打开浏览器跳转到 GitHub Release 页面（`html_url` 字段）
- **[稍后提醒]**：关闭弹窗，当次 App 运行内不再弹出（下次冷启动仍会检测）

**无新版本 / 检测失败**：静默处理，不展示任何 UI。

#### 2.9.5 隐私与合规

- 版本检测仅访问 GitHub 公开 API，不上传任何用户数据
- 检测请求中不携带设备标识、用户 ID 等个人信息
- 网络请求使用 HTTPS，不降级到 HTTP

#### 2.9.6 技术约束

| 项目 | 约束 |
|------|------|
| 网络库 | 使用 Android 原生 `HttpURLConnection` 或 `OkHttp`（如项目已引入）；**不**为此功能单独引入 Retrofit 等重型框架 |
| 检测超时 | 连接超时 5s，读取超时 5s；超时后静默失败 |
| 最小权限 | 仅需 `INTERNET` 权限（已在 Manifest 声明）|
| 检测频率 | 每次冷启动最多检测一次，不设后台周期任务 |
| 离线兼容 | 无网络时静默失败，不展示任何错误提示 |

#### 2.9.7 配置参数

以下参数通过 `BuildConfig` 或 `strings.xml` 配置，便于不同环境切换：

| 参数 | 示例值 | 说明 |
|------|--------|------|
| `UPDATE_CHECK_OWNER` | `your-github-username` | GitHub 仓库 owner |
| `UPDATE_CHECK_REPO` | `mushroom-adventure` | GitHub 仓库名 |
| `UPDATE_CHECK_ENABLED` | `true` | 是否启用版本检测（可在 debug 构建中关闭） |

### 2.10 跑酷小游戏模块（蘑菇大冒险 Run）

#### 2.10.1 功能概述

在 App 内嵌入一个横版无尽跑酷小游戏，以蘑菇为主角，玩法对标 Chrome 内置 Dino 游戏。游戏作为**每日全勤奖励**自动触发，每天只能游玩一次，游戏结束后分数自动保存到本地排行榜，以此激励学生完成每日学习任务。

#### 2.10.2 核心玩法

| 维度   | 设计                              |
| ---- | ------------------------------- |
| 视角   | 横版侧滚，角色在固定水平位置，背景向左滚动           |
| 操作   | **单指点击**屏幕 → 蘑菇跳跃（移动端无键盘，不支持俯身） |
| 死亡条件 | 蘑菇碰到任意障碍物                       |
| 结束方式 | 无限循环，无法通关，速度持续加快直到碰撞            |
| 重开限制 | **每日仅触发一次，不可重开**；游戏结束后自动保存分数并退出游戏页 |

#### 2.10.3 角色设计

**蘑菇主角（3 状态）**

| 状态  | 触发    | 视觉表现                     |
| --- | ----- | ------------------------ |
| 奔跑  | 游戏进行中 | 双腿交替动画（2 帧循环，每 150ms 切换） |
| 跳跃  | 点击屏幕  | 腾空抛物线运动，受重力下落            |
| 死亡  | 碰到障碍物 | 变红色 + 定格，不再运动            |

造型参考：圆形菌盖（红色带白点）+ 短腿，像素风格，与 App 内已有蘑菇图标保持视觉一致。

**跳跃物理参数**

| 参数      | 值            |
| ------- | ------------ |
| 初始跳跃速度  | `-18f`（向上）   |
| 重力加速度   | `0.8f`（每帧叠加） |
| 地面 Y 坐标 | 屏幕高度 × 0.75f |
| 二段跳     | 不支持          |

#### 2.10.4 障碍物设计

**地面障碍（始终存在）**

| 类型   | 尺寸          | 出现权重 |
| ---- | ----------- | ---- |
| 小石头  | 矮宽（40×30dp） | 40%  |
| 大石头  | 高宽（50×50dp） | 30%  |
| 单棵树桩 | 中等（30×50dp） | 20%  |
| 双连树桩 | 宽（70×50dp）  | 10%  |

**空中障碍（≥ 500 分出现）**

| 类型     | 飞行高度        | 应对方式         |
| ------ | ----------- | ------------ |
| 低空乌鸦   | 地面上方 80dp   | 跳跃越过         |
| 中/高空乌鸦 | 地面上方 150dp+ | 无需操作，蘑菇可直接跑过 |

> 说明：因移动端无俯身操作，中空/高空乌鸦设计为蘑菇可直接跑过，只有低空乌鸦需要跳跃，保持单点操作的简洁性。

#### 2.10.5 难度曲线

| 分数段         | 速度倍率    | 障碍最小间距   | 障碍种类      |
| ----------- | ------- | -------- | --------- |
| 0 – 200     | 1.0×    | 宽松（5s）   | 仅小石头      |
| 200 – 500   | 1.3×    | 正常（3.5s） | 地面障碍全种类   |
| 500 – 1000  | 1.6×    | 适中（3s）   | 地面 + 低空乌鸦 |
| 1000 – 2000 | 2.0×    | 紧凑（2.5s） | 混合随机      |
| 2000+       | 2.0× 渐进 | 挑战（2s）   | 混合，越来越密集  |

速度上限约为初始速度的 3×，防止画面完全不可操作。

#### 2.10.6 背景与视觉

| 元素   | 描述                                            |
| ---- | --------------------------------------------- |
| 地面   | 草地跑道线（绿色横线）                                   |
| 云朵   | 2～3 朵，随机速度缓慢向左飘动（视差：速度约为地面 20%）               |
| 远山   | 低速视差层（速度约为地面 10%），营造纵深感                       |
| 昼夜切换 | 每 700 分切换一次白天 ↔ 黄昏（背景色渐变，700 分白天，1400 分黄昏，循环） |

#### 2.10.7 计分系统

| 规则    | 细节                        |
| ----- | ------------------------- |
| 计分方式  | 存活时间换算，约每秒 +10 分，速度越快得分越快 |
| 里程提示  | 每 100 分：屏幕短暂闪白 + 提示音      |
| 分数上限  | 99999 分（理论上限，实际无法到达）      |
| 历史最高分 | 持久化存储到本地，跨会话保留            |

#### 2.10.8 积分排行榜

排行榜是本模块的核心差异化功能，记录玩家历次游戏的成绩，激励自我挑战与超越。

**排行榜数据结构**

| 字段    | 类型        | 说明       |
| ----- | --------- | -------- |
| rank  | Int       | 名次（1-10） |
| score | Int       | 本次得分     |
| date  | LocalDate | 游戏日期     |

**排行榜规则**

- 保留本地历史最高分记录，上限 **10 条**（Top 10）
- 每次游戏结束时，若分数进入 Top 10，自动插入并排序
- Game Over 页面**实时展示**是否创造新纪录（"🎉 新纪录！"）
- 主入口页面可通过"排行榜"按钮查看完整 Top 10 列表
- 排行榜本地存储（`SharedPreferences` 或 Room），**不联网**

**Game Over 页面布局**

```
┌─────────────────────────────────┐
│         GAME OVER               │
│                                  │
│    本次得分：  1 2 3 4           │
│    历史最高：  2 0 0 0           │
│                                  │
│    🎉 进入 Top 10！第 3 名       │  ← 仅在上榜时显示
│                                  │
│         [ 再玩一次 ]             │
│         [ 查看排行榜 ]           │
└─────────────────────────────────┘
```

**排行榜页面布局**

```

├────┬──────┬─────────────────────┤
│ #1 │ 2000 │  2026-03-06        │

```

#### 2.10.9 每日游玩奖励（蘑菇积分联动）

> 奖励采用**小蘑菇积分**方式（而非直接发放蘑菇），保持与主系统积分体系隔离，防止刷分。

| 触发条件           | 奖励     | 备注     |
| -------------- | ------ | ------ |
| 每日全勤触发游戏（完成所有任务） | 小蘑菇 ×1 | 鼓励完成每日任务 |
| 首次历史最高分突破 500  | 小蘑菇 ×1 | 一次性里程碑 |
| 首次历史最高分突破 1000 | 中蘑菇 ×1 | 一次性里程碑 |
| 首次历史最高分突破 2000 | 大蘑菇 ×1 | 一次性里程碑 |

**每日奖励上限**：每日游戏奖励上限为小蘑菇 ×1 + 里程碑奖励（里程碑仅触发一次）。

**设计原则**：游戏是完成当日任务的"彩蛋奖励"，每天只能玩一次、不可重开，从机制上确保游戏不会干扰正常学习节奏。

#### 2.10.10 游戏触发机制

**触发条件**：当天所有任务全部完成（即触发"全勤奖励"时）→ 系统自动弹出游戏入口提示，引导学生进入游戏。

**每日限定**：
- 每天只能玩一次，游戏结束后不可重开
- 若当天已玩过，全勤后的提示不再出现
- 排行榜和历史记录可随时查看

**游戏结束流程**：
1. 蘑菇碰撞障碍物 → 显示 Game Over 画面（分数、是否新纪录）
2. 停留 2 秒后自动退出游戏页，返回任务列表
3. 本次分数若进入 Top 10，自动插入排行榜

**游戏入口（排行榜查看）**：
- 任务页面常驻"排行榜"入口按钮，可随时查看历史成绩
- 若当天未触发游戏（任务未全部完成），入口仅可查看排行榜，不可开始游戏

#### 2.10.11 技术方案

| 模块     | 方案                                                   |
| ------ | ---------------------------------------------------- |
| 渲染引擎   | Jetpack Compose `Canvas` + `DrawScope`，绘制所有游戏元素      |
| 游戏循环   | `LaunchedEffect` + `withFrameMillis` 驱动每帧更新（约 60fps） |
| 状态机    | `GameViewModel`，状态：`IDLE` / `RUNNING` / `GAME_OVER`  |
| 物理模拟   | 简单抛物线重力模型（初速度 + 重力加速度叠加）                             |
| 碰撞检测   | AABB 包围盒检测，碰撞区域内缩 10%（避免边缘误判）                        |
| 输入处理   | `Modifier.pointerInput` 监听点击 → 触发跳跃                  |
| 音效     | `MediaPlayer` 播放跳跃/得分/碰撞音效（可静音）                      |
| 排行榜存储  | Room 数据库（新建 `GameScoreEntity` 表，最多保留 10 条）           |
| 蘑菇积分写入 | 调用现有 `MushroomRepository.recordTransaction()`        |
| 新模块    | `feature/feature-game`，独立模块，依赖 `core-domain`         |

### 3.1 性能需求
- 应用启动时间 < 2秒
- 页面切换流畅，无明显卡顿（60fps）
- 本地数据库查询响应 < 100ms

### 3.2 可靠性需求
- 本地数据持久化，不依赖网络连接（离线可用）
- 数据支持导出备份（JSON/CSV格式）
- 未来版本支持云端同步

### 3.3 安全需求
- 家长操作（审核打卡、奖品管理、成绩录入）需要密码/PIN验证
- 本地数据加密存储敏感配置

### 3.4 易用性需求
- 界面简洁直观，适合小学生使用
- 支持中文
- 蘑菇、拼图等关键元素使用可爱的视觉风格

### 3.5 扩展性需求
- 架构支持未来接入大模型 API（任务生成、学习建议）
- 支持多学科、多用户配置（兄弟姐妹共用同一设备）
- 奖励体系可完全自定义配置

### 3.6 版本管理需求
- 应用应能自动检测是否存在新版本，并在有新版本时以非打扰方式提示用户升级
- 版本检测不得阻塞应用启动或主流程，网络不可用时应静默失败
- 用户可选择"稍后提醒"，当次会话内不再重复弹出同一版本的提示
